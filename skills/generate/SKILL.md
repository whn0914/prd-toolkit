# PRD Generate

Guide a product manager through a structured conversation to produce a complete, reviewed PRD saved to `projects/`.

## Checklist

Complete in order:

1. **需求概述** — 主动询问需求概述，邀请提供原型截图
2. **探索项目背景** — 搜索当前目录下的相关文档，了解上下文
3. **多轮追问** — 逐轮提问，重点确认逻辑、合理性和完整性
4. **视觉伴侣（条件触发）** — 仅当原型+对话仍未澄清交互细节时才触发
5. **分章节展示 PRD 设计** — 逐节展示并获取确认
6. **写入 PRD 文档** — 生成并保存文件
7. **Subagent Spec Review** — 派发子 Agent 审查
8. **用户审阅确认** — 邀请用户确认并可选上传飞书

---

## Step 1: 需求概述

开场一句话：

> "请简单描述这个需求，如果有 Axure/Figma 原型截图可以直接粘贴（`Cmd+V`），我会结合原型和对话来整理需求。"

收到后：
- 如果是**多个独立小需求的合集**，逐一处理，不合并到同一 PRD
- 如果需求**范围过大**（横跨多个独立子系统），主动建议拆解：
  > "这个需求包含多个相互独立的功能，建议分开写 PRD。你希望先从哪个开始？"
- 合理范围的单一需求，继续 Step 2

---

## Step 2: 探索项目背景

在开始提问之前，先主动搜索当前工作目录：
- `projects/` 目录下同类功能的历史 PRD
- `.ai-config/` 下的模板和规范文件

如有相关背景信息，在后续提问中引用，避免重复询问已知内容。

---

## Step 3: 多轮追问

**核心原则：宁可多问，不要少问。目标是搞清楚需求，而不是尽快生成文档。**

规则：
- 每次只问一个问题，优先用选择题
- 收到原型截图时，立即分析提取交互逻辑，基于截图继续追问
- 必要时主动邀请：> "这里的交互逻辑还不够清晰，能补一张截图吗？"

重点追问方向：
- **逻辑合理性**：这个设计合理吗？有没有更简单的实现路径？
- **逻辑完整性**：A 触发后 B 怎么处理？边界情况是什么？存量数据怎么处理？
- **需求完整性**：这个场景覆盖了吗？有没有遗漏的用户路径？
- **生效版本**：每个需求点适用哪些版本（基础版/专业版/商务版）？
- **异常场景**：超时、数据为空、权限不足、并发时怎么处理？
- **数据与埋点**：哪些操作需要埋点，指标是什么？

对话满 10 轮后，可以先生成大纲草稿，边展示边补充细节。

---

## Step 4: 视觉伴侣（条件触发）

**判断：原型截图 + 已有对话是否已经把交互讲清楚了？**

- ✅ 已讲清楚 → **跳过，直接进 Step 5**
- ❌ 仍有模糊的布局/流程/交互细节 → 单独发送一条邀请消息（不混入其他内容）：

> "有一些交互细节用文字不太好描述，我可以在浏览器里展示流程图或布局对比来辅助确认。需要开启吗？（需要打开本地 URL）"

等待回复后再继续。

---

## Step 5: 分章节展示 PRD 设计

先展示大纲，确认后逐章节展示正文，复杂章节展示后询问是否需要调整。

**大纲格式：**
```
准备生成以下 PRD，请确认：

功能名称：xxx
核心目标：xxx
主要需求点：
  5.1 xxx（生效版本：专业版+）
  5.2 xxx（生效版本：全版本）
不在范围内：xxx
```

---

## Step 6: 写入 PRD 文档

按以下模板生成完整文档，保存到 `projects/[功能名称].md`（使用中文命名），`projects/` 不存在时自动创建。

```markdown
# [功能名称]

## 文档信息

| 角色 | 姓名 |
|------|------|
| 产品经理 | |
| UI | |
| 开发 | |
| 测试 | |

**原型蓝湖：**（待填写）
**蓝湖设计稿：**（待填写）
**发布版本：**（待填写）

---

## 变更说明

| 版本 | 变更人 | 变更日期 | 变更内容 |
|------|-------|---------|---------｜
| 1.0.0 | | [TODAY] | [一句话概述本次需求] |

---

## 一、业务背景与价值点

**背景：**
[需求来源和用户痛点]

**价值点：**
1. [核心价值]

---

## 二、目标与范围

**目标：**
1. [目标]

**不在范围内：**
- [明确排除项]

---

## 三、目标用户

[用户类型、使用场景、核心诉求]

---

## 四、流程

**[流程名称]：**
[分步骤文字描述]

---

## 五、产品/交互逻辑

### [需求点名称]

**生效版本：** [基础版 / 专业版 / 商务版]

[交互规则和约束]

---

## 六、异常流程与边界条件

| 场景 | 触发条件 | 系统处理方式 |
|------|---------|------------|
| 网络超时 | | |
| 数据为空 | | |
| 权限不足 | | |
| 并发操作 | | |

---

## 七、验收标准

- [ ] [具体可测试的验收条件]

---

## 八、数据与埋点需求

- [事件名]：[采集内容和目的]

---

## 九、遗留问题

| 问题 | 负责人 | 截止日期 |
|------|-------|---------|
| | | |
```

**字段归属：**

| 字段 | 填写方 |
|------|--------|
| 所有内容章节 | AI（基于对话） |
| 变更说明 | AI（自动：v1.0.0，今日日期，一句话概述） |
| 产品经理/UI/开发/测试/蓝湖/发布版本 | 保留占位符 |

**异常流程必须覆盖：** 网络超时、数据为空/缺失、权限不足、并发/重复操作。

**验收标准要求：** 每条必须具体可测试，禁止出现"功能正常"、"展示正确"等模糊表述。

---

## Step 7: Subagent Spec Review

生成文档后，读取同目录下的 `prd-reviewer-prompt.md`，将完整 PRD 内容作为上下文，派发一个 Agent 子任务执行审查。

审查重点（覆盖以下四个维度）：
1. **逻辑完整性**：流程是否有断点？边界情况是否遗漏？
2. **逻辑合理性**：设计是否存在自相矛盾或不合理假设？
3. **需求完整度**：9 个章节是否实质性填写？验收标准是否可测试？
4. **上下文一致性**：各章节之间（目标、流程、逻辑、验收标准）是否相互呼应，有无冲突？

如审查发现问题：修复后重新派发（最多 2 轮）。2 轮后仍有问题的，以 `⚠️ 存疑：[内容]` 标注在文档中。

---

## Step 8: 用户审阅确认

> "PRD 已生成并完成质检，文件保存在 `projects/[文件名].md`。
> 请确认内容是否符合预期。如需修改请告知，确认后可以帮你上传到飞书。"

如用户说"上传飞书"或"导入飞书"，使用 `docx_builtin_import` 将 markdown 文件内容导入飞书，返回飞书文档链接。
